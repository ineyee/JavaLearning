## 一、Redis 的内存数据淘汰策略

> 实际开发中，在配置文件里配置一下即可，详见第一篇

我们知道 Redis 的数据是存储在服务器内存中的，但是整台服务器的内存又不一定是只供 Redis 使用，所以我们就得给 Redis 分配一个它能使用的最大服务器内存，比如 1GB：

| 项目规模 |    建议分配     |
| :------: | :-------------: |
|   小型   |  256mb ~ 512mb  |
|   中型   |    1gb ~ 2gb    |
|   大型   | 4gb+ 或单独评估 |

当我们给 Redis 分配了最大内存限制后，就得考虑一个问题，那就是内存数据淘汰策略，因为随着数据的增加绝对会超出最大内存限制，超出后应该告诉 Redis 该怎么丢数据：

> LRU means Least Recently Used、最后一次**使用时间** -> 长时间没使用的优先淘汰
> LFU means Least Frequently Used、历史**使用次数** -> 使用总次数少的优先淘汰

|        淘汰策略        |       淘汰范围       | 淘汰算法 |                         说明                          |
| :--------------------: | :------------------: | :------: | :---------------------------------------------------: |
| **noeviction（默认）** |          -           |    -     |   **内存满了直接抛内存溢出异常，拒绝后续的写操作**    |
|      volatile-lru      | 设置了过期时间的 key |   LRU    |   在设置了过期时间的 key 里，长时间没使用的优先淘汰   |
|    **allkeys-lru**     |       所有 key       |   LRU    |       **在所有 key 里，长时间没使用的优先淘汰**       |
|      volatile-lfu      | 设置了过期时间的 key |   LFU    |   在设置了过期时间的 key 里，使用总次数少的优先淘汰   |
|    **allkeys-lfu**     |       所有 key       |   LFU    |       **在所有 key 里，使用总次数少的优先淘汰**       |
|    volatile-random     | 设置了过期时间的 key |   随机   |          在设置了过期时间的 key 里，随机淘汰          |
|     allkeys-random     |       所有 key       |   随机   |                在所有 key 里，随机淘汰                |
|      volatile-ttl      | 设置了过期时间的 key |   TTL    | 在设置了过期时间的 key 里，剩余存活时间最短的优先淘汰 |

## 二、Redis 的系统方案

#### 1、扩展方案

假设我们有一台 Redis 服务器，当用户量上来以后，这台服务器可能就扛不住了，为了让这台服务器能抗住，我们可以做两件事情：

* 给这台服务器加内存、加硬盘、加 CPU，升级单台服务器的配置我们称之为**垂直扩展**
* 直接加服务器搞成服务器集群，加服务器我们称之为**水平扩展**

#### 2、数据可靠性方案

> 实际开发中，在配置文件里配置一下即可，详见第一篇

Redis 的数据是存储在内存中，所以读写速度非常快，但是一旦服务器出现宕机，内存中的数据就会丢失，所以 Redis 还提供了**持久化机制（RDB 快照、AOF 日志）来将内存中的数据异步地持久化到硬盘**，以便在服务器宕机重启后能恢复数据。

###### 2.1 RDB 快照（Redis Database 快照）

* ① RDB 快照是什么？

RDB 快照是指每隔一段时间就**自动**把当前时间点 Redis 内存中的所有**数据**压缩成一个文件持久化到硬盘，文件名通常是 dbfilename = dump.rdb、文件路径通常是 $dir/$dbfilename。这样一来当服务器重启后就可以从硬盘上读取快照文件，解压后恢复到内存中

* ② 自动生成快照策略怎么配置？

时间间隔在配置文件里配置一下即可，满足条件后 Redis 会自动生成快照并持久化，建议的快照生成策略：

```
# 如果 15 分钟内发生了 1 次写操作，就生成快照持久化一次
save 900 1
# 或者如果 5 分钟内发生了 10 次写操作，就生成快照持久化一次
save 300 10
# 或者如果 60 秒内发生了 10000 次写操作，就生成快照持久化一次
save 60 10000
```

* ③ 所有数据是写在一个快照文件里吗？是的，每次生成快照都会生成一个新的快照文件，并且永远只会有一个快照文件、不会分文件
* ④ 那下次生成快照文件后，旧快照文件会被覆盖吗？是的，旧快照文件会被新快照文件覆盖
* **⑤ RDB 快照的优点是什么？写入磁盘的频率没那么高、所以 IO 性能开销较低；文件增长速度相对较慢、所以最终文件相对较小，将来数据恢复速度快**
* **⑥ RDB 快照的缺点是什么？比如说我们设置了上面的快照生成策略，但是 290 秒内只发生了 9 次写操作，此时服务器突然宕机了，那么就不会生成新快照，那这 9 次写操作的数据就丢失了，也就是说两次快照之间的数据可能丢失**

###### 2.2 AOF 日志（Append Only File 日志）

* ① AOF 日志是什么

AOF 日志是指 Redis 每执行一条或多条命令就**自动**把**命令**追加到硬盘上的日志文件里，Reids6 之前的文件名通常是 appendfilename = appendonly.aof、文件路径通常是 $dir/$dbfilename，Redis6 之后的文件目录通常是 appenddirname = appendonlydir、目录路径通常是 $dir/$appenddirname。这样一来当服务器重启后就可以从硬盘上读取日志文件，重新执行文件里的所有命令把数据恢复到内存中

* ② 自动追加命令策略怎么配置？

```
# 每执行一条命令都追加到日志里，不会丢数据，所以最安全
# IO 性能开销大
# 实际开发中一般不用
appendfsync always
```

```
# 每执行一条命令先放到命令缓存中，然后每秒把缓存中的命令追加到日志里，顶多丢 1 秒的数据，安全性可接受
# IO 性能开销相比 appendfsync always 低很多
# 实际开发中最常见
appendfsync everysec
```

```
# 每执行一条命令先放到命令缓存中，何时把缓存中的命令追加到日志里由操作系统决定，可能丢几十秒的数据，安全性最差
# IO 性能开销最低
# 实际开发中一般不用
appendfsync no
```

* ③ 日志文件也是一个文件吗？会分文件吗？Redis7 之前只有一个日志文件，Redis7 之后有多个日志文件

```
# Redis7 之前的配置写法
appendonly yes
appendfilename "appendonly.aof"
```

```
# Redis7 之前的配置写法（我们只要配置目录即可，日志文件由 Redis 自动维护）
appendonly yes
appenddirname "appendonlydir"
```

* **④ AOF 快照的优点是什么？数据丢失更少，更安全**
* **⑤ AOF 快照的缺点是什么？写入磁盘的频率较高、所以 IO 性能开销较高；文件增长速度相对较快、所以最终文件相对较大，将来数据恢复速度慢**

###### 2.3 RDB + AOF

实际开发中可以把两种方案都配上去，都配的话会优先使用 AOF、因为 AOF 的数据更全，RDB 作为兜底

#### 3、高可用方案

> 一个高可用系统要达到以下要求：
>
> * 要能避免单点故障问题
> * 要能提高吞吐量 QPS
> * 数据量上来以后要能存得下

###### 3.1 主从服务器 + 哨兵机制

假设我们只有一台 Redis 服务器，那么一旦这台服务器挂掉，整个系统就运转不起来了，这就是单点故障问题。所以我们可以**搞一台 Redis 主服务器、再搞一台或多台 Redis 从服务器，这就是主从服务器，这样一来一旦主服务器挂掉了，Redis 的哨兵机制组件就会检测到，它就会从多台从服务器里选举出一台来作为新的主服务器，并自动完成主从切换，**成功避免单点故障问题。

此外主从服务器还有以下好处：

* 从服务器负责实时备份主服务器上的数据，从而避免数据丢失

* 主服务器只负责写数据操作，从服务器只负责读数据操作，实现读写分离，从而提高“读数据”的吞吐量、但是不能提高“写数据”的吞吐量

###### 3.2 主从服务器分片集群

主从服务器能避免单点故障问题，但是不能提高“写数据”的吞吐量（读数据的确是分流到各个从服务器了，但是写数据还是一台主服务器），也无法保证数据量上来以后能存得下（从服务器只是起到数据备份的作用，而不是起到数据分开存储的作用，也就是说数据还是要完整存一份在主服务器上）。所以我们可以**搞成多主多从，这就是主从服务器分片集群，这样一来写数据也可以分流到不同的主服务器，数据可以分片存储到不同的主服务器、而不是必须完整一份存储到某台主服务器，**成功解决提高吞吐量问题和数据量上来以后要能存得下问题。
